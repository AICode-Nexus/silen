# Silen 无模型 AI 质量闭环设计

- 状态：等待书面设计稿评审
- 日期：2026-07-22
- 仓库：`AICode-Nexus/silen`
- 包：`@aicode-nexus/silen`

## 1. 概述

Silen 将新增一套确定性的文档质量闭环。即使用户没有 AI 模型、模型提供商账号、API Key、托管端点或网络连接，这套闭环仍可正常工作。它会评测真实用户问题能否检索到预期的已发布页面或章节，提供可执行的失败报告，并可作为本地或 CI 发布门禁。

首个版本将新增 `silen ai eval`，修正 `silen ai audit` 对部署 `base` 的链接处理，并停止把可选的本地工作区缓存作为只读搜索的必要条件。评测将直接使用生产文档站实际使用的 `search-index.json`，而不是另一套平行的检索实现。

Ask AI 仍是可选的端点适配器。这套质量闭环不会使用 Ask AI；未配置端点时，Silen 也不会把它呈现为已经可用的助手。

## 2. 产品决策

选定方案是一套离线检索质量门禁：

1. 作者提交一份小型、不可执行的代表性问题评测集。
2. 常规 Silen 构建生成生产搜索索引。
3. `silen ai audit` 验证源文件和生成产物的完整性。
4. `silen ai eval` 使用生产搜索索引运行所有问题。
5. 如果预期页面或章节没有出现在允许的前若干条结果中，CI 失败。

这是主要支持的 AI 就绪路径，不是 Ask AI 的降级替代品。没有模型的站点必须保留完整的构建、审计、评测、报告、本地搜索、AI 产物以及只读 MCP 体验。

## 3. 目标

首个版本必须：

1. 在没有模型、嵌入、模型提供商 SDK、API Key、端点或网络访问的情况下工作。
2. 评测与站点读者实际使用的词法搜索行为。
3. 允许作者在提交的 JSON 中描述“问题到页面”以及可选的“问题到章节”预期。
4. 生成确定性的人工可读报告和机器可读报告。
5. 返回适用于 CI 的稳定退出码。
6. 为每个失败用例展示实际排序结果。
7. 修复部署在非根路径 `base` 下的站点产生的错误断链报告。
8. 保持 MCP 预检只读，并且不执行工作区配置。
9. 将 `.silen/ai/index.json` 定位为可选的显式缓存，而非审计前置条件。
10. 将 Silen 官方双语站点作为可运行的评测示例。
11. 保持现有 Ask AI 行为：未配置端点时，不生成 Ask AI 控件，也不生成 Ask AI 客户端 bundle。

## 4. 非目标

首个版本不会：

- 使用模型生成答案、问题、预期路由或修复文本。
- 添加嵌入、向量数据库、重排序或语义搜索。
- 托管或代理 Ask AI 服务。
- 添加模型提供商预设、凭据、遥测或远程 MCP。
- 根据绝对分数阈值调优检索。
- 自动改写文档、评测用例或配置。
- 执行 Git commit、push、部署或其他 shell 工作流。
- 让 `silen build` 更新作者编写的评测文件或本地 AI 缓存。
- 使用 `ai-index.json` 或原始工作区搜索代替生产读者搜索索引进行评测。

## 5. 无模型产品契约

无模型路径是一项第一等产品契约：

```text
silen build <root>
  -> silen ai audit <root>
  -> silen ai eval <root>
  -> deterministic report and exit code
```

即使不存在与模型提供商相关的环境变量，且网络不可用，这三个命令仍然可以使用。`ai eval` 只读取评测集和已构建的搜索索引。它不会导入 Ask AI 客户端、调用 `fetch`、检查模型配置，也不要求配置 `themeConfig.ai`。

Ask AI 仍是一项独立的可选运行时增强能力。配置端点后可以添加现有主题控件，但不会改变审计或评测行为。端点不可用时，只能影响当前 Ask AI 请求；绝不能影响构建、本地搜索、审计、评测、生成的 AI 产物或 MCP。

## 6. 用户工作流

### 6.1 建立本地质量基线

作者创建 `.silen/ai-evals.json`，构建站点，然后运行：

```sh
pnpm silen build docs
pnpm silen ai audit docs
pnpm silen ai eval docs
```

只有当每个预期页面或章节都出现在配置的 Top K 结果中时，评测才通过。

### 6.2 在 CI 中使用相同检查

CI 在不配置模型密钥的情况下运行相同的三个命令。检索回归会返回非零状态，并输出足够的排序证据，以便在本地复现。整个过程不需要专门的托管评测服务。

### 6.3 诊断失败问题

对于每个失败项，报告会展示：

- 用例标识和查询。
- 预期路由和可选章节标题。
- 实际生效的语言和 Top K 设置。
- 实际排序后的路由、章节标题、语言和诊断分数。
- 有边界的修复提示：改进相关标题、描述、章节标题或页面正文，或者修正作者编写的预期。

工具不会选择或应用修复。

## 7. 架构

### 7.1 事实来源边界

质量闭环包含三类相互独立的来源：

1. 作者编写的内容和 `.silen/ai-evals.json` 是需要评审的源文件。
2. `.silen/dist/search-index.json` 是生成的生产检索状态。
3. `.silen/ai/index.json` 是可选且可重建的本地工作区缓存。

只有生产搜索索引是评测目标。这样可以避免评测虽然通过，却掩盖读者实际搜索体验中的回归。本地工作区缓存仍可通过 `silen ai index` 使用，但它的状态不决定内存内 MCP 搜索或生产站点是否可用。

### 7.2 评测集

默认评测集路径是内容根目录中的 `.silen/ai-evals.json`。它使用纯 JSON，因此加载时不会执行项目代码。

版本 1 的 Schema 如下：

```json
{
  "schemaVersion": 1,
  "topK": 5,
  "cases": [
    {
      "id": "model-free-ai",
      "query": "Can Silen work without an AI model?",
      "lang": "en-US",
      "expected": {
        "route": "/ai/",
        "heading": "Public AI artifacts"
      }
    },
    {
      "id": "zh-read-only-mcp",
      "query": "MCP 默认有写入权限吗？",
      "lang": "zh-CN",
      "expected": {
        "route": "/zh/ai/local-workspace-mcp/"
      }
    }
  ]
}
```

验证规则刻意保持精简和严格：

- `schemaVersion` 必须为 `1`。
- `topK` 可选，默认为 `5`，并且必须是 1 到 20 之间的整数。
- `cases` 必须包含 1 到 500 个条目。
- 每个 `id` 均不能为空，长度不得超过 100 个字符，并且必须唯一。
- 每个 `query` 在空白字符归一化后必须包含 1 到 500 个字符。
- `lang` 可选，并使用现有的搜索语言偏好机制。
- `expected.route` 是不包含部署 base、且以 `/` 开头的站点路由。
- `expected.heading` 可选；存在时不能为空。
- 拒绝未知字段，避免拼写错误的预期被静默接受。
- 评测集文件大小上限为 1 MiB。

路由不包含部署 base。预期路由和实际路由都要进行索引路由等价归一化，包括尾部斜杠和 `index` 形式。章节标题预期在空白字符归一化后采用不区分大小写的精确文本匹配。章节标题和路由必须同时出现在同一条排序结果中。

### 7.3 评测运行器

`silen ai eval [root]` 执行以下步骤：

1. 解析内容根目录，不执行 `.silen/config.ts`。
2. 读取并验证 `.silen/ai-evals.json`。
3. 读取并验证 `.silen/dist/search-index.json`，确认其为当前支持的搜索索引版本。
4. 通过共享的 Node 侧生产搜索管线运行每个查询。
5. 应用现有的可选语言偏好，并取前 `topK` 条结果。
6. 匹配预期路由以及存在时的章节标题。
7. 收集所有结果后，再渲染一份稳定报告。

现有公开搜索结果的数据结构保持兼容。共享的内部排序查询辅助方法保留 MiniSearch 分数，用于评测诊断；面向站点的查询函数继续返回当前结果结构。分数以一致规则舍入后展示，绝不作为通过条件。

结果顺序保持为确定性搜索并列排序规则产生的顺序。用例顺序保持为评测集中的编写顺序。运行器不会在第一个失败用例后停止。

### 7.4 感知 base 的审计

审计接收归一化后的 `base` 上下文，不再假设每个根相对链接都从 `/` 开始。

直接运行 `silen ai audit` 时，按以下顺序解析 base：

1. 存在且有效的已构建站点 Agent Contract 清单。
2. 可信项目配置，执行边界与直接调用 `silen build` 相同。
3. 只有当前两者都无法提供 base 时才使用 `/`，并同时生成一个非阻断的 base 未知提示。

MCP 的 `build` 预检和 MCP 审计路径绝不执行项目配置。它们可以读取已有的构建清单；否则使用 `/`，并报告无法验证 base。

对于 `/silen/` 这样的已配置 base，`/silen/guide/` 这样的作者链接要与编译器路由 `/guide` 比较。只移除一次完全匹配且已归一化的 base。`/silen-other/` 这样的相似前缀不会被移除。相对链接、仅包含 fragment 的链接、外部 URL 和协议相对 URL 保持现有处理方式。

### 7.5 审计发现项的严重级别

`issues` 继续作为控制 `ok` 的阻断问题集合。审计新增非阻断的 `notices` 集合，用于提供对操作人员有帮助、但不能证明站点损坏的状态信息。

`.silen/ai/index.json` 缺失或过期时，生成一条 `index-cache` 提示。提示说明可以运行 `silen ai index` 刷新可选快照，同时内存内 MCP 搜索仍然可用。缺少生产产物、契约失败、断链和无效引用仍是阻断问题。

对于此前把未使用的缓存视为发布失败的 CI，这是一次有意的兼容性行为变更。`silen ai index` 仍需显式运行；审计和构建都不会写入该缓存。

### 7.6 报告

默认输出是简洁的人工可读报告。`--json` 输出一个带版本的 JSON 文档，标准输出中不包含其他说明文字。

JSON 结果包含：

```json
{
  "schemaVersion": 1,
  "ok": false,
  "suite": ".silen/ai-evals.json",
  "index": ".silen/dist/search-index.json",
  "topK": 5,
  "summary": { "total": 2, "passed": 1, "failed": 1 },
  "cases": [
    {
      "id": "model-free-ai",
      "ok": false,
      "query": "Can Silen work without an AI model?",
      "lang": "en-US",
      "expected": { "route": "/ai/" },
      "actual": [
        {
          "rank": 1,
          "route": "/integrations/",
          "title": "Integrations",
          "heading": "Ask AI",
          "lang": "en-US",
          "score": 12.345678
        }
      ]
    }
  ]
}
```

报告不包含时间戳、耗时、绝对文件系统路径、模型提供商状态、环境变量值或随机标识符。固定的键构造方式、用例顺序、排序结果顺序和分数舍入规则，保证相同输入生成逐字节一致的 JSON。

## 8. 错误与退出行为

CLI 使用以下退出状态：

- `0`：评测集有效，并且所有用例通过。
- `1`：评测集成功运行，但一个或多个用例失败。
- `2`：由于输入、Schema、索引或文件系统状态无效，评测无法运行。

可预期的运行前置错误包括：

- 缺少评测集：指出 `.silen/ai-evals.json`，并展示最小创建示例。
- 缺少搜索索引：指出 `.silen/dist/search-index.json`，并提示用户运行 `silen build <root>`。
- JSON 无效：报告文件和解析错误，不暴露无关的环境或文件系统信息。
- Schema 无效：能够确定用例 ID 时报告该 ID，并指出精确的 JSON 字段路径。
- 索引版本不受支持：报告发现的版本，并提示用户使用已安装的 Silen 版本重新构建。
- 没有搜索结果：记录空的 `actual` 列表并判定该用例失败，而不是把它视为运行器错误。

在 JSON 模式下，可预期的运行前置错误也会输出一个结构化错误文档，然后以 `2` 退出。意外的内部错误继续由常规 CLI 错误边界处理，并返回非零状态。

审计仍然只在存在阻断问题时返回 `1`。提示项不会改变退出状态。

## 9. 安全与副作用

`ai eval` 是只读的。它不得：

- 创建 `.silen`、输出文件、缓存、日志或临时文件。
- 加载可执行的站点配置或 MDX 模块。
- 通过路径穿越或符号链接读取内容根目录之外的文件。
- 访问模型提供商凭据或环境特定的模型设置。
- 发起网络请求。
- 修改源内容、评测集或 Git 工作区。

评测集和构建索引采用与现有 AI 工作区读取相同的有界、符号链接安全文件处理原则。直接 CLI 审计只可为了获取 `base` 等站点事实而加载可信项目配置；MCP 预检继续保持更严格的“不执行配置”边界。

## 10. 验证策略

### 10.1 单元测试覆盖

新增聚焦以下场景的测试：

- 有效的“仅路由”与“路由加章节标题”用例。
- 默认和显式 `topK`。
- 英文和中文语言偏好。
- 路由、尾部斜杠和章节标题归一化。
- 空结果，以及正确结果位于允许排名之外的情况。
- 稳定排序和分数舍入。
- 缺失、超大、格式错误、不支持、为空、ID 重复、含未知字段、路由无效和限制无效的输入。
- 生产搜索当前接受的索引版本。
- 评测期间不写入、不联网，也不依赖 Ask AI。

### 10.2 审计回归覆盖

新增测试证明：

- 在 base 为 `/silen/` 时，`/silen/guide/` 解析为 `/guide`。
- 根 base 站点保持现有行为。
- 在 `/silen/` 下不会错误移除 `/silen-other/guide/`。
- 相对、外部、fragment、编码和格式错误的目标继续得到安全处理。
- 本地缓存过期时生成提示；只要所有阻断检查通过，`ok` 仍为 true。
- MCP 预检在解析审计上下文时不会执行恶意的 `.silen/config.ts`。

### 10.3 CLI 与集成测试覆盖

新增 CLI 测试证明：

- `ai eval` 出现在命令契约和帮助文本中。
- 通过、检索失败和运行前置错误的状态分别为 `0`、`1` 和 `2`。
- 人工可读输出包含有用的失败摘要和实际 Top K 结果。
- `--json` 输出能够被解析为单个稳定文档。
- 不存在模型提供商环境变量时，命令仍能成功。
- 运行命令不会改变源文件或生成文件。

Silen 官方站点提交双语用例，并且必须通过：

```sh
corepack pnpm site:build
node dist/node/cli.js ai audit website
node dist/node/cli.js ai eval website
```

仓库还必须通过类型检查、聚焦测试、完整测试套件、lint、格式检查以及发布流程已要求的包构建检查。

## 11. 文档变更

英文和中文的 AI 工作区、CLI 与参考页面将新增：

- `ai eval` 命令和评测集路径。
- 最小双语评测集示例。
- 明确说明不需要模型、API Key、端点或网络。
- 说明生产搜索索引与可选本地工作区缓存的区别。
- 三种退出状态及 CI 用法。
- 保持现有边界：未配置端点时不启用 Ask AI。

措辞不得暗示 Silen 内置了对话助手。

## 12. 兼容性与迁移

现有站点只有在调用 `silen ai eval` 时才需要评测集。现有构建、本地搜索、AI 产物、MCP 命令和 Ask AI 端点集成保持兼容。

审计 JSON 结果新增非阻断提示。消费者应继续使用 `ok` 和 `issues` 作为发布门禁。`.silen/ai/index.json` 过期时不再令 `ok` 变为 false；需要该快照的用户可以继续显式运行 `silen ai index`。

评测 Schema 和 JSON 报告各自携带版本号，使未来变更能够显式失败，而不是静默改变语义。

## 13. 已考虑的替代方案

### 13.1 提供真正的托管 Ask AI 服务

本阶段不采用，因为它需要选择模型和基础设施，并处理凭据、成本控制、隐私政策、滥用防护和网络可用性。它也无法满足没有 AI 模型的用户。

### 13.2 要求本地模型或嵌入运行时

不采用，因为安装体积、硬件差异、模型许可和非确定性输出会削弱零配置 CI 门禁。

### 13.3 评测 `ai-index.json`

不采用，因为它是 AI 发现产物，而不是生产搜索界面使用的检索实现。

### 13.4 评测原始的内存内 MCP 工作区搜索

不将其作为主要门禁，因为它采用不同的文档模型和排序路径。它仍适用于有边界的本地智能体读取，但通过这项评测并不能证明读者搜索质量。

### 13.5 使用 MiniSearch 分数作为门禁

不采用，因为绝对分数会随语料库变化。基于排名的路由和可选章节标题预期能够更持久地表达用户可见需求；分数只保留用于诊断。

## 14. 交付边界

只有满足以下条件，实施阶段才算完成：

1. 感知 base 的审计消除官方站点当前部署前缀造成的误报，同时不会掩盖相似但错误的路由。
2. 可选工作区缓存的状态不再阻断审计。
3. `silen ai eval` 实现严格、带版本的评测集和报告契约。
4. 官方双语站点提供能够通过的无模型评测用例。
5. 所有无模型、确定性、安全、CLI、审计和回归测试均通过。
6. 文档将无模型路径描述为默认且完整的能力，并将 Ask AI 描述为可选能力。

模型集成、生成式回答、语义检索和自动内容修复都需要单独设计并获得批准。
